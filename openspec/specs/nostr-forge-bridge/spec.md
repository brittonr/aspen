## ADDED Requirements

### Requirement: Forge push generates NIP-34 repo state event

The bridge plugin SHALL subscribe to Forge hook events for ref updates and generate a `kind:30618` (repository state announcement) Nostr event containing the current ref heads.

#### Scenario: Push to main branch

- **WHEN** a git push updates `refs/heads/main` on a Forge repository
- **THEN** the plugin SHALL emit a `kind:30618` event with a `d` tag matching the repo identifier, `refs/heads/main` tag with the new commit ID, and `HEAD` tag pointing to the default branch

#### Scenario: Multiple ref updates in single push

- **WHEN** a git push updates multiple refs (e.g., `main` and a tag `v1.0`)
- **THEN** the plugin SHALL emit a single `kind:30618` event containing all updated ref tags

### Requirement: Forge repo creation generates NIP-34 repo announcement

The bridge plugin SHALL generate a `kind:30617` (repository announcement) Nostr event when a new Forge repository is created.

#### Scenario: New repository created

- **WHEN** a new Forge repository is created with name "my-project" and description "A test project"
- **THEN** the plugin SHALL emit a `kind:30617` event with `d` tag (repo identifier), `name` tag, `description` tag, and `maintainers` tag listing the repo delegates

#### Scenario: Repo announcement includes clone URL

- **WHEN** a repository announcement is generated
- **THEN** the event SHALL include a `clone` tag with the `aspen://` clone URL for the repository

### Requirement: Forge issues generate NIP-34 issue events

The bridge plugin SHALL generate `kind:1621` (issue) Nostr events when issues are created on Forge repositories.

#### Scenario: Issue created

- **WHEN** a new issue is created on a Forge repository with subject "Bug in auth" and body "Login fails..."
- **THEN** the plugin SHALL emit a `kind:1621` event with content set to the issue body, an `a` tag referencing the repository announcement, a `p` tag for the repo owner, and a `subject` tag with the issue title

#### Scenario: Issue with labels

- **WHEN** an issue is created with labels ["bug", "auth"]
- **THEN** the event SHALL include `t` tags for each label

### Requirement: Forge patches generate NIP-34 patch events

The bridge plugin SHALL generate `kind:1617` (patch) Nostr events when patches are submitted to Forge repositories.

#### Scenario: Single patch submitted

- **WHEN** a patch is submitted to a Forge repository
- **THEN** the plugin SHALL emit a `kind:1617` event with content set to the patch diff (git format-patch output), an `a` tag referencing the repository announcement, a `p` tag for the repo owner, and a `t` tag "root" if it is the first patch in a series

### Requirement: Bridge events signed with cluster Nostr key

All Nostr events generated by the bridge plugin SHALL be signed with the cluster's secp256k1 Nostr key. The `pubkey` field SHALL be the cluster's Nostr public key.

#### Scenario: Event signature valid

- **WHEN** a Nostr client receives an event from the bridge
- **THEN** the event's secp256k1 Schnorr signature SHALL verify successfully against the `pubkey` field

### Requirement: Bridge activates via hook subscriptions

The bridge plugin SHALL subscribe to Forge-related hook topics to receive events. The plugin SHALL NOT poll or scan the KV store.

#### Scenario: Hook-driven activation

- **WHEN** a Forge repository push occurs
- **THEN** the hook system SHALL deliver the event to the bridge plugin, which SHALL process it and emit the corresponding Nostr event

#### Scenario: No Forge activity

- **WHEN** no Forge events occur
- **THEN** the bridge plugin SHALL consume zero CPU and emit no events
